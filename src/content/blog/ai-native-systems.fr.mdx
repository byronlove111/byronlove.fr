Les agences tournent sur 15 outils différents. CRM ici. Docs là. Tracking dans Notion. Factures ailleurs. Chaque système détient un fragment de la vérité. C'était acceptable quand un humain faisait la synthèse.

Un agent ne peut pas. Pas efficacement.

Si ton agent a besoin de 8 appels API pour comprendre qui est le client, ce qu'il veut, et ce qui s'est passé jusqu'ici — ton système n'est pas prêt pour l'IA. Il est AI-tolérant. Ce n'est pas la même chose. Et ça compte plus que le modèle que tu utilises.

## Le diagnostic

Le problème n'est pas l'absence d'outils. La plupart des équipes ont déjà un LLM quelque part. Le problème, c'est ce qui se passe quand ce LLM doit agir.

Un backend d'agence classique ressemble à ça : un CRM pour les contacts et les deals, un outil de gestion de projet pour les tâches, une plateforme docs pour les briefs clients, un outil séparé pour le tracking des campagnes, les factures dans encore autre chose. Chaque système parle aux humains. Les humains ouvrent quatre onglets, synthétisent, décident.

Quand tu mets un agent dans cet environnement, il hérite du même morcellement. Il tape 8 endpoints. Il assemble un tableau fragmenté depuis des schémas qui se recoupent à moitié. Il prend des décisions sur un contexte qu'il a dû reconstruire lui-même — et quand il hallucine ou fait une erreur, tu ne sais pas quel fragment a causé le problème.

C'est ça, AI-tolérant : le système fonctionne techniquement avec de l'IA greffée dessus, mais il n'a jamais été conçu pour. Chaque appel API supplémentaire, c'est de la latence, du coût, et une occasion de rater quelque chose. Chaque schéma incompatible est une taxe de prompt engineering que tu paies indéfiniment.

## Le changement de mental model

Dès que tu décides de construire AI-native, une question remplace toutes les autres : *qu'est-ce qu'un agent a besoin pour agir ici, et comment je lui donne ça en un seul coup ?*

Pas "quelles sont mes entités ?" Pas "quelles sont mes routes REST ?" Ce sont de bonnes questions pour un produit destiné aux humains. Pour un agent, c'est le mauvais point de départ.

Un agent qui travaille sur une mission client a besoin de : qui est le client, quels sont ses objectifs, quelles campagnes tournent, quelles ont été les 20 dernières tâches, quels documents sont attachés, et quels outils sont disponibles. C'est un objet de contexte. Pas huit endpoints — une réponse, structurée pour la façon dont un agent pense.

```typescript
// AI-tolérant : l'agent reconstruit le contexte depuis 8 endpoints
const company   = await api.get('/companies/:id')
const contacts  = await api.get('/companies/:id/contacts')
const deals     = await api.get('/companies/:id/deals')
const campaigns = await api.get('/campaigns?company=:id')

const documents = await api.get('/documents?company=:id')
const invoices  = await api.get('/invoices?company=:id')
const history   = await api.get('/activity?company=:id')
// 8 appels. 8 occasions d'avoir un contexte incomplet. Pas de contrat de schéma clair.

// AI-native : un endpoint de contexte taillé pour l'agent
const context = await api.get('/agent-context/mission/:id')
// {
//   company:         { name, icp, industry, size, health_score },
//   contacts:        [{ name, role, last_interaction }],
//   mission:         { brief, kpis, type, deadline },
//   active_tasks:    [{ id, title, status, assigned_to }],
//   completed_tasks: [...20 dernières, avec leurs outputs],
//   documents:       [{ name, type, url }],
//   tools_available: ['emelia', 'github', 'slack', 'brevo']
// }
// Un appel. Image complète. L'agent peut démarrer immédiatement.
```

Ça change aussi ce que tu optimises. Avec des APIs fragmentées, tu optimises chaque endpoint indépendamment. Avec un endpoint de contexte, tu optimises la latence agent — combien de temps entre "tâche reçue" et "j'ai tout ce qu'il me faut pour agir" ?

## MCP, pas juste du REST

La couche d'accès compte aussi. Le REST fonctionne, mais le MCP — Model Context Protocol — est fait pour les agents. Au lieu que ton agent appelle des endpoints, tu exposes un serveur MCP. L'agent se connecte, déclare ce dont il a besoin, ton système retourne des données structurées sémantiquement riches.

Le gain concret : n'importe quel agent — Cursor local, agent cloud, Claude qui tourne sur le laptop de quelqu'un — peut se brancher sans intégration sur mesure à chaque fois. Tu écris le protocole une fois. Tous les modèles le parlent.

Mais le MCP est aussi un forcing function. Il t'oblige à penser en termes de *ressources* et d'*outils* plutôt qu'en entités et routes. Une ressource, c'est ce que l'agent lit pour comprendre le contexte. Un outil, c'est ce qu'il appelle pour agir. Cette distinction est utile : elle sépare ce que l'agent doit comprendre de ce qu'il doit faire, et tu conçois chacun différemment.

> Si tu construis un nouveau backend en 2026 et que ton premier réflexe est encore du CRUD REST, pause — et demande-toi qui appelle vraiment ces endpoints, et s'ils ont besoin de l'image complète ou juste d'un fragment.

La question n'est pas REST contre MCP. C'est : est-ce que je conçois pour un humain qui navigue dans une UI, ou pour un agent qui doit agir de façon autonome avec le moins d'aller-retours possible ? La réponse te dit tout sur comment structurer ton système.

## Le sandbox et le monitoring ne sont pas optionnels

Dès que les agents agissent — pas juste génèrent du texte, mais appellent des outils, écrivent du code, envoient des messages, mettent à jour des données — le sandboxing et le monitoring deviennent de l'infrastructure portante.

Chaque run doit être isolé. L'agent voit exactement ce que tu lui donnes. Il n'appelle que les outils que tu as autorisés. Son output arrive dans une file de review avant que quoi que ce soit d'irréversible se passe. Tu as un log complet : ce que l'agent a vu, ce qu'il a décidé, ce qu'il a appelé, ce qui a changé.

Sans ça, débugger c'est de l'archéologie. Quelque chose a mal tourné en prod — quel run ? Quel contexte avait-il ? Quel outil a causé la cascade ? Si tu ne peux pas répondre à ces questions en moins d'une minute, ton monitoring n'est pas conçu pour les agents.

L'autre raison de sandboxer : ça rend l'itération peu coûteuse. Tu peux relancer un agent sur le même contexte gelé, changer un prompt, voir ce qui change. Tu peux rejouer des cas d'échec. Tu peux tester un nouveau modèle sur des tâches historiques avant de le promouvoir en prod. Rien de tout ça n'est possible si chaque run mute de l'état en direct.

## S37 Portal — ce que ça veut dire concrètement de construire AI-native

On construit ça from scratch chez S37. On est une agence — missions GTM, design, dev pour des clients réels. Le contexte de chaque client était éparpillé entre Attio, Notion, des fils d'emails, Google Drive, et la mémoire de qui avait participé au dernier call.

Donc on a tout reconstruit autour d'un seul principe : une source de vérité unique, structurée pour les agents dès le premier jour.

**L'architecture a trois couches.**

La première : la source de vérité. Toutes les données de l'agence en un seul endroit. Companies, contacts, deals, missions, tâches, documents, factures. Ce qui était éparpillé dans quatre outils vit maintenant dans un seul schéma, conçu pour être lu par des agents — pas juste navigué par des humains.

La deuxième : l'infrastructure d'agents. Chaque run démarre avec un appel API qui retourne le contexte complet de la mission — company, ICP, toutes les tâches actives, les tâches complétées avec leurs outputs, les documents attachés, les outils disponibles. L'agent ne cherche pas. Il reçoit. Ensuite, chaque type de mission a son environnement d'exécution adapté avec les bons outils câblés.

La troisième : l'interface — un dashboard pour l'équipe, un portail client pour les clients. Mêmes données, deux vues.

**Missions GTM — Alexis**

Alexis crée une tâche avec un brief, choisit le modèle, clique Run. L'agent reçoit le contexte complet de la campagne : stats Emelia des 3 derniers mois, définition de l'ICP, séquences précédentes qui ont performé, background de la company. Il ne demande rien. C'était dans le contexte.

Il analyse ce qui convertit, identifie les points de friction, génère une recommandation avec la séquence suivante proposée. Alexis review, ajuste si nécessaire, approuve. L'itération qui prenait une demi-journée prend vingt minutes — pas parce que l'IA est plus intelligente, mais parce qu'elle avait l'image complète dès le départ.

Pour une mission complète, un agent orchestrateur prend le relais. Il lit toutes les tâches, identifie les dépendances, lance en parallèle ce qui peut l'être. Analyse ICP, génération de séquence, qualification de la base, lancement Emelia — en quelques minutes, sans que personne soit derrière son clavier.

**Missions Design — Zinedine**

Zinedine travaille en code. Son agent reçoit le brief, clone le repo, implémente le composant en respectant le design system, lance le projet, prend des screenshots du rendu, et ouvre une PR avec les screenshots attachés.

Zinedine voit le résultat visuel sans avoir à checkout la branche. Il review directement dans Portal, approuve, ou envoie du feedback. L'agent itère. Toute la boucle se passe dans Portal — sans switcher entre GitHub, Figma, et un environnement de dev local.

**Missions Dev — Byron**

Même boucle pour le dev. L'agent reçoit le contexte, clone le repo du client, implémente la feature, fait tourner les tests, ouvre une PR. Byron review le diff et les résultats de tests dans Portal. Veut des modifications ? Il envoie du feedback dans Portal. L'agent re-run.

Le travail humain devient : définir l'intention clairement, review l'output, décider. L'exécution est déléguée.

**Le portail client**

Chaque client a accès à son espace sur portal.s37.studio. Il voit ses missions actives, les tâches qu'Alexis a choisi d'exposer, ses documents, ses factures. Il peut uploader des fichiers, signer son contrat, télécharger ses rapports.

Et il peut poser des questions à un agent qui a accès à ses données — cloisonnées à sa company seulement. "Où en sont mes campagnes ce mois-ci ?" L'agent interroge la base, retourne les stats, compare aux objectifs. Sans qu'Alexis ait envoyé un seul message.

L'impression d'avoir un account manager dédié disponible 24/7. La réalité : un agent, un contexte scopé, aucun travail supplémentaire pour l'équipe.

## Ce qui devient possible à l'échelle

Une fois l'infrastructure en place, la roadmap s'ouvre.

**Monitoring proactif.** Un agent tourne en fond toutes les heures, vérifie les stats Emelia de toutes les missions actives, détecte les baisses de performance, crée automatiquement une tâche "investiguer la campagne en sous-performance" et prévient Alexis. Personne n'a à se souvenir de vérifier. Le système remonte ce qui a besoin d'attention.

**Mémoire cross-missions.** L'agent sait que pour Copwell, l'ICP qui convertit le mieux ce sont les DSI en Île-de-France dans des boîtes de 200-500 personnes. Il réutilise ça automatiquement pour les nouvelles bases sans qu'Alexis ait à le redire. Le contexte s'accumule entre les missions au lieu de repartir de zéro à chaque fois.

**Planification de mission.** Alexis crée la company, attache le transcript du call et le brief ICP. Un agent lit tout, propose un plan de mission complet : tâches, KPIs cibles, séquences Emelia suggérées, timeline. Alexis valide et ajuste, les agents commencent à exécuter. De l'onboarding à l'exécution en une session.

Rien de tout ça n'est de la science-fiction. Chaque pièce est techniquement faisable aujourd'hui. Ce qui bloque la plupart des équipes, ce n'est pas la capacité des modèles — c'est que leurs systèmes n'ont pas été conçus pour donner aux agents le contexte et les outils dont ils ont besoin pour agir.

## Ce qu'il faut changer concrètement aujourd'hui

Trois décisions qui se cumulent :

**1. Conçois des endpoints de contexte, pas des endpoints d'entités.**
Arrête de construire `GET /companies/:id` + `GET /deals` + `GET /contacts`. Construis `GET /agent-context/mission/:id`. La différence, c'est pour qui tu conçois — un humain qui clique dans une UI, ou un agent qui doit agir.

**2. Pense MCP quand tu construis des outils que les agents vont réutiliser.**
Surtout pour les intégrations : Emelia, GitHub, Slack, facturation. Tu l'écris une fois, tu l'exposes à tous les modèles. Arrête de reconstruire les intégrations cas par cas.

**3. Sandbox chaque run dès le premier jour.**
Isolation totale, log complet, file de review avant tout ce qui est irréversible. Ce n'est pas de l'overhead — c'est comme ça que tu déploies des agents sans appréhension, et que tu les améliores de façon systématique dans le temps.

## Ce qu'il faut retenir

Le bottleneck n'est plus l'intelligence du modèle. Claude et GPT peuvent exécuter la majorité de ce que ton équipe fait. Le bottleneck, c'est l'environnement : contexte fragmenté, pas de sandbox, pas de monitoring, 8 appels pour répondre à une question simple sur un client.

Les anciens systèmes étaient conçus pour des humains qui naviguent entre des onglets. Les agents ne naviguent pas entre des onglets. Ils ont besoin de l'image complète, vite, structurée, en un seul endroit.

Les équipes qui construisent pour ça — aujourd'hui, from scratch — auront un avantage structurel très difficile à rattraper plus tard. Pas parce que leur IA est meilleure. Parce que leur système l'est.
