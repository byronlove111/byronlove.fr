Agencies run on 15 different tools. CRM here. Docs there. Tracking in Notion. Invoices somewhere else. Each system holds one slice of the truth. That was fine when humans connected the dots.

Agents can't. Not efficiently.

If your agent needs 8 API calls to understand who a client is, what they want, and what's already happened — your system is not ready for AI. It's AI-tolerant. There's a difference, and it matters more than which model you're running.

## The diagnosis

The problem isn't new tooling. Most teams already have LLMs somewhere. The problem is what happens when those LLMs need to act.

A typical agency backend looks like this: a CRM for contacts and deals, a project management tool for tasks, a docs platform for client briefs, a separate tool for campaign tracking, invoices in yet another place. Every system talks to humans. Humans open four tabs, synthesize, decide.

When you put an agent in that environment, it inherits the same fragmentation. It hammers 8 endpoints. It pieces together a picture from half-matching schemas. It makes decisions on context it partially assembled — and when it hallucinates or makes a wrong call, you have no idea which fragment of context caused it.

This is AI-tolerant: the system technically works with AI bolted on, but it was never designed for it. Every extra API call is latency, cost, and a chance to miss something. Every schema mismatch is a prompt engineering tax you pay indefinitely.

## The mental model shift

The moment you decide to build AI-native, one question replaces all the others: *what does an agent need to act on this, and how do I give it that in one shot?*

Not "what are my entities?" Not "what are my REST routes?" Those are fine questions for a human-facing product. For an agent, they're the wrong starting point.

An agent working on a client mission needs: who the client is, what their objectives are, what campaigns are running, what the last 20 tasks were, what documents are attached, and what tools are available. That's one context object. Not eight endpoints — one response, structured for how an agent thinks.

This also changes what you optimize for. With fragmented APIs, you optimize each endpoint independently. With a context endpoint, you optimize for agent latency — how fast can the agent go from "task received" to "I have everything I need to act"?

## MCP, not just REST

The access layer matters too. REST works, but MCP — Model Context Protocol — is built for agents. Instead of your agent calling endpoints, you expose an MCP server. The agent connects, declares what it needs, and your system returns structured semantically-rich data.

The practical win: any agent — local Cursor, cloud agent, Claude running on someone's laptop — can connect to your system without a custom integration each time. You write the protocol once. Every model speaks it.

But MCP is also a forcing function. It makes you think in terms of *resources* and *tools* instead of entities and routes. A resource is something the agent reads for context. A tool is something the agent calls to take action. That distinction is useful: it separates what the agent needs to understand from what it needs to do, and you design each differently.

> If you're building a new backend in 2026 and your first instinct is still REST CRUD, pause and ask who's actually calling those endpoints — and whether they need the full picture or just one slice.

The question isn't REST vs MCP. It's: am I designing for a human navigating a UI, or for an agent that needs to act autonomously with minimal round trips? The answer tells you everything about how to structure your system.

## Sandboxing and monitoring aren't optional

Once agents act — not just generate text, but actually call tools, write code, send messages, update records — sandboxing and monitoring become load-bearing infrastructure.

Every agent run should be isolated. The agent sees exactly what you give it. It calls only the tools you've authorized. Its output lands in a review queue before anything irreversible happens. You have a full log: what the agent saw, what it decided, what it called, what changed.

Without this, debugging is archaeology. Something went wrong in production — which agent run? Which context did it have? Which tool call caused the cascade? If you can't answer those questions in under a minute, your monitoring isn't built for agents.

The other reason to sandbox: it makes iteration cheap. You can re-run an agent on the same frozen context, change a prompt, see what changes. You can replay failure cases. You can test a new model on historical tasks before promoting it. None of that is possible if every run mutates live state.

## S37 Portal — what building AI-native actually looks like

We're building this from scratch at S37. We're an agency — GTM, design, dev missions for real clients. Every client's context was spread across Attio, Notion, email threads, Google Drive, and whoever remembered the last conversation.

So we rebuilt around a single principle: one source of truth, structured for agents from day one.

**The architecture has three layers.**

The first is the source of truth: all agency data in one place. Companies, contacts, deals, missions, tasks, documents, invoices. What was scattered across four tools now lives in one schema, designed to be read by agents, not just navigated by humans.

The second is the agent infrastructure. Each agent run starts with one API call that returns the full mission context — company, ICP, all active tasks, completed tasks with their outputs, attached documents, available tools. The agent doesn't search. It receives. From there, each mission type has its own execution environment with the right tools wired in.

The third is the interface — a dashboard for the team, a client portal for clients. Same data, two views.

**GTM missions (Alexis)**

Alexis creates a task with a brief, picks the model, clicks Run. The agent receives the full campaign context: Emelia stats for the last 3 months, ICP definition, previous sequences that performed, company background. It doesn't ask for any of it. It was in the context.

From there, it analyzes what's converting, identifies drop-off points, generates a recommendation with a proposed next sequence. Alexis reviews, adjusts if needed, approves. The iteration that used to take half a day takes twenty minutes — not because the AI is smarter, but because it had the full picture from the start.

For a complete mission, an orchestrator agent takes over. It reads all tasks, identifies dependencies, launches what can run in parallel. ICP analysis, sequence generation, list qualification, Emelia launch — in a few minutes, without anyone behind a keyboard.

**Design missions (Zinedine)**

Zinedine works in code. His agent receives the brief, clones the repo, implements the component respecting the design system, spins up the project, takes screenshots of the rendered output, then opens a PR with the screenshots attached.

Zinedine sees the visual result without checking out the branch. He reviews directly in Portal, approves, or sends feedback. The agent iterates. The whole loop happens in Portal — no switching between GitHub, Figma, and a local dev environment.

**Dev missions (Byron)**

Same loop for dev. Agent gets the context, clones the client's repo, implements the feature, runs the tests, opens a PR. Byron reviews the diff and test results in Portal. Wants changes? He sends feedback in Portal. The agent re-runs.

The human work becomes: define the intent clearly, review the output, decide. The execution is delegated.

**The client portal**

Every client has access to their own space at portal.s37.studio. They see their active missions, tasks Alexis has chosen to expose, their documents, their invoices. They can upload files, sign contracts, download reports.

And they can ask questions to an agent that has access to their data — scoped to their company only. "Where are my campaigns this month?" The agent queries the database, returns stats, compares against objectives. Without Alexis having sent a single message.

The impression of having a dedicated account manager available 24/7. The reality: one agent, scoped context, no extra work from the team.

## What becomes possible at scale

Once the infrastructure works, the roadmap opens up.

**Proactive monitoring.** An agent runs in the background every hour, checks Emelia stats across all active missions, detects performance drops, automatically creates a task "investigate campaign underperformance" and pings Alexis. No one has to remember to check. The system surfaces what needs attention.

**Cross-mission memory.** The agent knows that for Copwell, the ICP that converts best is IT directors in Île-de-France at 200-500 person companies. It reuses that automatically for new lists without Alexis having to say it again. Context accumulates across missions instead of starting from zero each time.

**Mission planning.** Alexis creates the company, attaches the call transcript and ICP brief. An agent reads everything, proposes a full mission plan: tasks, KPI targets, suggested Emelia sequences, timeline. Alexis validates and adjusts, and agents start executing. From onboarding to execution in one session.

None of this is science fiction. Every piece is technically tractable today. What blocks most teams isn't model capability — it's that their systems weren't built to give agents the context and tools they need to act.

## What to actually change today

Three decisions that compound:

**1. Design context endpoints, not entity endpoints.**
Stop building `GET /companies/:id` + `GET /deals` + `GET /contacts`. Build `GET /agent-context/mission/:id`. The difference is who you're designing for — a human clicking through a UI, or an agent that needs to act.

**2. Think MCP when building tools agents reuse across missions.**
Especially for integrations: Emelia, GitHub, Slack, billing. Write the tool once, expose it to every model. Stop rebuilding integrations per-use-case.

**3. Sandbox every run from day one.**
Isolated execution, full audit log, review queue before anything irreversible. This isn't overhead — it's how you ship agents without fear, and how you improve them systematically over time.

## The point

The bottleneck isn't model intelligence. Claude and GPT can execute most of what your team does. The bottleneck is the environment: fragmented context, no sandbox, no monitoring, 8 calls to answer a simple question about a client.

Old systems were designed for humans navigating tabs. Agents don't navigate tabs. They need the full picture, fast, structured, in one place.

The teams that build for that — today, from scratch — will have a structural advantage that's very hard to close later. Not because their AI is better. Because their system is.
