Alexis, notre GTM, a une idée. Il m'envoie un message Discord. Je le lis deux jours plus tard, je pose des questions, on s'aligne, je développe, je déploie. Feature live en fin de semaine si tout va bien.

C'est pas catastrophique. Mais c'est moi le problème.

<AnimatedSection>

## Le vrai goulot

Dans une petite agence, le dev est souvent le seul à pouvoir traduire une intention business en code qui tourne en prod. Résultat : tout passe par lui. Idée de design, ajout de feature, correction de copie — tout finit dans sa queue.

On a un stack solide : trois repos (site vitrine, outil de propositions, automations), Railway qui auto-déploie sur push, Discord pour communiquer. La mécanique de déploiement était déjà là. Le problème était humain.

</AnimatedSection>

<AnimatedSection>

## Le shift : MCP-first

On a tous accès à Claude Code. Alexis inclus.

Au lieu de construire des bots custom et des interfaces spéciales pour "simplifier les choses pour les non-tech", on a réalisé qu'on pouvait juste brancher les bons outils ensemble.

## Ce qu'on a branché

**Linear MCP** — Alexis décrit son idée à Claude Code. Claude génère un brief structuré (titre, acceptance criteria, repo concerné, complexité estimée) et crée le ticket Linear directement via MCP. Pas de Git, pas de terminal, pas de friction.

**Linear → Discord** — intégration native de Linear. Quand un ticket est créé ou change de statut, Byron reçoit une notification Discord. Zéro code custom.

**Cursor Background Agent** — une fois le ticket approuvé, un agent lit le brief, comprend le contexte grâce à l'`AGENTS.md` de chaque repo, implémente la feature et ouvre une PR. Byron review le code, merge.

**Railway auto-deploy** — sur merge vers main, Railway déploie automatiquement. C'était déjà en place. On ne le voyait juste pas comme une pièce du puzzle.

</AnimatedSection>

<AnimatedSection>

## Ce qu'on n'a pas construit

Pas de bot Discord custom. Pas d'interface "feature request" spéciale. Pas de webhook handler pour parser des messages.

Le piège classique : vouloir coder la glue entre les outils alors que la glue existe déjà. MCP-first, ça veut dire chercher l'intégration native avant d'écrire du code.

</AnimatedSection>

<AnimatedSection>

## Le vrai takeaway

L'ère agentic ne remplace pas les devs. Elle supprime les frictions entre les rôles.

Alexis ne devient pas dev. Il devient capable d'exprimer une intention dans un format qu'un agent peut exécuter. C'est suffisant. Et ça change complètement la vélocité d'une petite équipe.

La question à se poser : dans ton workflow actuel, qui joue le rôle de traducteur humain ? Probablement toi. Et probablement, tu pourrais l'automatiser.

</AnimatedSection>
