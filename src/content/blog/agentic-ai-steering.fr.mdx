Sur quasi toutes les feuilles de route, une case à cocher « agentique » est apparue. Rarement elle précise qui a le droit de faire quoi, pendant combien de temps et sans quel niveau de supervision. Pourtant tout est là.

Dire qu'un système est « agentique » fusionne habituellement trois idées : **il peut enchaîner plusieurs actions**, **il peut utiliser des outils externes**, et **il peut boucler tant qu'il croit avoir terminé**. Utile ? Souvent. Sûr et aligné avec le produit d'office ? Non.

## L'autonomie a besoin de limite et d'arbitrage

Les agents séduisent parce qu'ils déchargent la planification. Quelqu'un doit quand même trancher trois choses :

1. **L'intention** — ce que signifie le succès, en critères vérifiables plutôt qu'en slogan.
2. **Le périmètre d'autorisation** — quels outils, quelles données, quels montants, quels dépôts, quelles suppressions sont permis pendant une exécution.
3. **Les conditions d'arrêt** — temps max, jalons avant effets irréversibles, escalade humaine lorsque la confiance baisse.

En manquer une, ce n'est plus un cadre pour agent : c'est un script déguisé en intelligence ou, pis, un stagiaire aléatoire avec des jetons OAuth.

> La difficulté n'est pas d'autoriser une étape de plus dans le dialogue. C'est de décider quelles étapes ne doivent jamais passer sans validation.

Pensez en **politiques** : liste blanche d'API, quotas de nouvel essai, bac à sable tant qu'un contrôle automatique ou humain n'a pas dit oui.

## Mieux vaut observer que se fier au récit

Une boucle agent saine ressemble davantage à une chaîne de contrôle-qu' à une conversation gratuite. Quelque chose **observe** le réel (logs, linter, API de facturation, boîte mail), quelque chose **planifie ou modifie**, quelque chose **décide** si la boucle continue.

Si la vérification, c'est « le modèle a dit que ça a marché », la prod vous rapprendra vite les limites du récit. Ce qui compte, c'est la vérité terrain : tests en échec, plafond de taille de diff, alertes métier, file d'escalade.

<CodeBlock client:load lang="text">
Intention → Plan (profondeur bornée) → Action (outils autorisés) → Vérification (hors LLM) → Revue si échec ou faible confiance → Arrêt
</CodeBlock>

Pas besoin d'un schéma pour chaque prototype, mais les équipes qui sautent la colonne « vérifier » livrent souvent des régressions en habit de progrès.

## Le message pour la direction

Les agents sont un levier : ils accélèrent autant les bons changements que le rayon des catastrophes. Le pilotage est un travail produit continu — resserrer les politiques après incident, reformuler les critères de succès après des entretiens utilisateurs, retirer des outils trop dangereux.

Si votre organisation ne peut pas dire **qui dort mal quand le robot déraille**, mieux vaut faire une pause avant d'exposer vos clients à une boucle encore plus large sans filet humain.
